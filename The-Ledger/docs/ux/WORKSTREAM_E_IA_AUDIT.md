# 1. PURPOSE

Workstreams A–D each own a *role*. Workstream E owns the *seams between them*.

Its goal is singular:

> The Ledger must feel like one coherent product, not a collection of independent modules.

E therefore audits nothing that A–D already own. It audits only what no single
workstream could see: the inconsistencies that are invisible inside one module and
obvious when a user crosses between two.

---

# 2. AUDIT METHOD

The audit was conducted against the merged state of all four role workstreams:

| Surface | Entry point | Files |
|---|---|---|
| CEO / PM (internal) | `client/src/components/layout.tsx` | 48 pages |
| Worker (mobile) | `client/src/components/WorkerMobileLayout.tsx` | 8 pages |
| Client (portal) | `client/src/components/portal/PortalShell.tsx` | 8 pages |

Three shells. Three independently-developed presentation vocabularies. Every finding
below is a divergence between them, or a divergence within one of them.

---

# 3. DOCTRINE POSITION

**Workstream E is a presentation-layer consolidation only.**

This is the same posture as UX-5 (Intelligence Hub). E makes no change to:

- Any engine in `client/src/lib/`
- The Client Portal projection layer (`portalProjections.ts`)
- RBAC role gates or route protection
- The Review Centre approval path
- Any audit event, its emission point, or its payload
- Any financial normalization logic

E changes what things are *called* and what they *look like*. It does not change what
they *do*. Doctrine impact is nil by construction; E-6 proves this by test rather than
by assertion.

# 4. FINDINGS

## F-1 — Terminology is split between British and American spelling — CRITICAL

The platform uses "Centre" and "Center" interchangeably in user-facing copy.

The worst instance is a live branch on role, in `client/src/pages/review.tsx:442`:

```
{isCEO ? "Review Operations Centre" : "Review Center"}
```

**The same destination is named differently depending on who is looking at it.** A CEO
and a PM discussing the same screen are using different words for it.

Full inventory of American-spelling user-facing copy:

| File | Line | String |
|---|---|---|
| `pages/review.tsx` | 442 | `Review Center` (PM branch) |
| `pages/review-detail.tsx` | 43 | `Back to Review Center` |
| `pages/qa.tsx` | 35 | `QA Center` |
| `pages/financial-explorer.tsx` | 95 | `…generated by the Review Center…` |
| `pages/payroll.tsx` | 74, 152, 233 | `Review Center` ×3 |
| `pages/payroll-export.tsx` | 164, 254 | `Review Center` ×2 |
| `components/JobIntelligenceSection.tsx` | 202 | `Review Center` |
| `components/finance/InvoiceReadinessPanel.tsx` | 49 | `Review Center` |
| `components/finance/JobForecastPanel.tsx` | 225 | `Review Center` |
| `components/finance/PendingExposurePanel.tsx` | 52 | `Review Center` |

Against these, the platform elsewhere consistently uses **Centre**: Reporting Centre,
Analytics Centre, Reconciliation Centre, Exception Resolution Centre, Automation Centre,
Notification Centre, Workflow Centre, Executive Command Centre, Review Operations Centre.

**Verdict:** Centre is the established majority form and matches the platform's UK
domain language (Labour, Programme, Normalisation). Center is the exception and is
eliminated.

### F-1a — Test coupling assessment

Critical to E-2's risk profile, and verified before scoping:

- `tests/helpers/navigation.ts` → `openReviewCenter()` selects `getByTestId('nav-review')`.
  It is a *function name*, not a text assertion.
- **No test in the suite asserts the literal text `Review Center` or `QA Center`.**
- 108 test references to `Centre` already exist and remain valid, because the sweep only
  moves Center → Centre, never the reverse.

**E-2 therefore carries no test-breakage risk.** The internal helper name is retained
as-is to avoid churn in 4 spec files for zero user-visible benefit.

## F-2 — No page-header primitive; heading hierarchy is broken — HIGH

> **CORRECTION (post-implementation).** As first written, this finding claimed that
> "roughly 30 of 48 internal pages render no `<h1>` anywhere on the page." **That was
> wrong.** It came from grepping `client/src/pages/*.tsx` without checking the `Layout`
> wrapper that every page renders inside. `layout.tsx:410` rendered
> `<h1>The Ledger</h1>` in the sidebar, so every page always had exactly one `h1`.
>
> The real defect was different, and narrower: that single `h1` was the **application
> name**, not the page title. It told a screen-reader user what the product is called,
> never what page they were on — and it was inconsistent with the mobile header, which
> already rendered the brand as a `div`.
>
> The correction cost real time. Adopting `PageHeader` initially produced a *second*
> `h1` on every page, which failed 25 of the new IA tests until `layout.tsx` was fixed
> to render the brand mark as a `div`. Recorded here because the audit was the input to
> every later phase, and an unverified premise propagated straight through it.

33 pages hand-roll their page title. There is no `PageHeader` component.

Two consequences:

**Accessibility defect.** The only `<h1>` on any page was the sidebar brand mark. Page
titles rendered as `<h2>`, so the document outline began at level 2 and no heading
described the current page. Examples: `audit.tsx:13`, `clients.tsx:77`, `jobs.tsx:253`,
`schedule.tsx:271`, `roles.tsx:85`, `review.tsx:246`, `automations.tsx:1267`.

**Visual inconsistency.** Where an `h1` does exist, size is unpredictable:

| Size | Pages |
|---|---|
| `text-3xl` | accounting-settings, expenses, integrations |
| `text-2xl` | analytics-centre, executive-command-centre, finance-hub, intelligence-hub, reporting-centre, notification-center |

The hubs (2xl) and the leaf pages (3xl) disagree, so title size shifts as the user
navigates between them.

## F-3 — Hardcoded colour literals bypass design tokens — HIGH

40 files use raw Tailwind palette values (`text-slate-900`, `text-gray-900`,
`bg-slate-50`, `text-slate-500`) instead of the semantic tokens the theme defines
(`foreground`, `muted-foreground`, `card`, `border`).

Highest concentrations:

| File | Occurrences |
|---|---|
| `pages/review-detail.tsx` | 30 |
| `pages/invoice-detail.tsx` | 19 |
| `pages/review.tsx` | 14 |
| `pages/asset-detail.tsx` | 13 |
| `pages/location-detail.tsx` | 13 |
| `pages/stock-detail.tsx` | 13 |
| `components/dev/SynchronizationDebugPanel.tsx` | 12 |
| `pages/invoices.tsx` | 11 |

This is **latent, not cosmetic**. The values are near-invisible in light mode, which is
why they survived four workstreams. In dark mode they render dark-on-dark. Any future
theming or white-labelling work — which the Client Portal already promises under
"Owns Trust → White-labelling" — is blocked until these are tokenised.

## F-4 — The Client Portal is more polished than the executive platform — MEDIUM

Workstream D built a proper `EmptyState` component (`pages/portal/finance.tsx:581`) with
icon, title, body and a stable testid, used consistently across portal sections:

```
<EmptyState testid="portal-quotes-empty" icon={FileSignature}
            title="No quotes" body="Quotes issued to you will appear here." />
```

The internal platform has **one** ad-hoc local equivalent (`financial-explorer.tsx:48`,
label-only, no icon) and bare inline strings everywhere else.

The commercial implication is uncomfortable: the surface shown to *customers* is more
finished than the surface shown to the *buyer*. Investor and demo audiences see the
executive platform first.

## F-5 — Navigation labels do not match their destinations — MEDIUM

| Nav label | Destination | Page actually titled |
|---|---|---|
| Expenses | `/expenses` | "Financial Insights" |
| Stock & Assets | `/equipment` | "Stock & Assets" |
| Accounting Settings | `/finance?tab=accounting` | Finance Hub, Accounting tab |
| Review *(CEO)* | `/review` | "Review Operations Centre" |
| Reviews *(PM)* | `/review` | "Review Center" |

"Expenses → Financial Insights" is the clearest defect: the label promises a list of
expenses and delivers an analytics page. The Review row is F-1 seen from the nav side.

Polish**, rather than allowing them to expand E-4. Without this boundary, "finish
Workstream E" has no definable completion criterion.

# 6. FROZEN PLATFORM LEXICON

Authoritative for all four role surfaces. E-2 enforces; E-6 tests.

## 6.1 Spelling

**British English throughout.** Centre, Programme, Normalisation, Labour, Organisation,
Prioritisation, Utilisation.

Sole exception: identifiers already frozen in code — file names
(`reconciliation-center.tsx`), route paths, `data-testid` values, engine function names,
and the `openReviewCenter` test helper. **These are not user-facing and are not changed
by E**, because renaming them risks 915 passing tests for no user benefit.

> **Rule:** the lexicon governs strings a user can read. It does not govern identifiers.

## 6.2 Canonical destination names

| Canonical name | Route | Visible to |
|---|---|---|
| Review Centre | `/review` | CEO, PM |
| Review Operations Centre | `/review` | CEO *(hub-mode title only)* |
| Finance Hub | `/finance` | CEO |
| Intelligence Hub | `/intelligence` | CEO |
| Automation Centre | `/automations` | CEO |
| Workflow Centre | `/workflows` | CEO |
| Notification Centre | `/notifications` | PM |
| Reconciliation Centre | `/finance?tab=accounting` | CEO |
| Exception Resolution Centre | `/finance?tab=accounting` | CEO |
| Client Requests | `/client-requests` | CEO, PM |
| QA Centre | `/qa` | CEO, PM |

**Doctrine note.** "Review Centre" is the canonical name of the control mechanism in
every doctrine document. Body copy referring to the approval gate must say *Review
Centre* — never *Review Center*, never *review queue*, never *approvals*. The doctrine
and the interface must use the same word for the same thing.

## 6.3 Role-invariant terminology

The same concept keeps the same word across CEO, PM, Worker and Client surfaces.
Where a role may not see a concept at all, that is an RBAC matter, not a naming one.

| Concept | Canonical term | Never |
|---|---|---|
| A unit of work | Job | Project *(internal)*, Task, Ticket |
| A unit of work *(Client Portal)* | Project | Job |
| A person doing the work | Worker | Employee, Staff, Resource |
| A group of workers | Crew | Team, Gang |
| Submitted, not yet approved | Pending Review | Awaiting, In Queue, Unapproved |
| Approved and normalised | Approved | Signed off, Confirmed |
| Rejected back to submitter | Rejected | Denied, Declined *(reserved for Client Requests)* |
| Money owed to us | Outstanding | Unpaid, Owing, Receivable |
| Unapproved financial estimate | Pending Exposure | Forecast, Projected cost |
| Advisory projection | Projection — Advisory Only | Forecast *(unqualified)* |

The Job / Project split is **deliberate and doctrinally required**: the Client Portal
speaks the client's language, and `CLIENT_PORTAL_DOMAIN.md` names the entity Project.
This is the one permitted cross-role divergence, and it is enforced by the projection
layer rather than by copy.

## 7.1 Risk control for E-4

E-4 touches 33 pages against a suite of 915 tests that select heavily on `data-testid`.

Controls:

1. `PageHeader` accepts a `testId` prop. It **never generates** one. Every existing
   testid is carried across verbatim.
2. The full set of `data-testid` values is extracted before and after each E-4 commit
   and diffed. A non-empty removal set fails the phase.
3. Heading *text* is unchanged by E-4 — only the element, size and structure. Text
   changes belong to E-2 and land in a separate commit, so a failure is attributable to
   exactly one phase.

## 7.2 Verification

Per repository owner's standing preference, Claude verifies with `npm run build`; the
Playwright suite is run locally by the owner.

Because E-2 and E-4 are precisely the phases where a build-only check is weakest, E-6
will produce an explicit list of every spec file touched and every locator affected, so
that the owner's run is a checklist rather than a discovery exercise.

Expected baseline after E: **915 existing + new `platform-ia.spec.ts` tests, 0
regressions.**

# 8. SUCCESS CRITERIA

Workstream E is complete when:

1. No user-facing string uses American spelling of a canonical term.
2. Every page renders exactly one `<h1>`, via `PageHeader`, at a consistent size.
3. No page or component file uses a raw palette literal for text or surface colour.
4. Empty states use one shared component across all four role surfaces.
5. Every navigation label matches the title of the page it opens.
6. `platform-ia.spec.ts` enforces 1–5 so they cannot silently regress.
7. Build PASS; full suite green with 0 regressions against the 915 baseline.
