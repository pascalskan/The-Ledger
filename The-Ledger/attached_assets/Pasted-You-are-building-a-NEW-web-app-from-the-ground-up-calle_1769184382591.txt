You are building a NEW web app from the ground up called “The Ledger” for Omnisoftware. 
IMPORTANT: Do NOT reuse or copy my existing Ledger codebase. Treat it ONLY as a proof-of-concept reference. Start clean.

Goal:
Build a production-ready trade-business operations management tool that helps businesses manage Workers, Jobs, and Equipment (Assets), including scheduling assignments, conflict detection, audit logs, and an interactive jobs map.

Non-negotiables:
- Every button must work end-to-end (no placeholders). If there is an “Add Worker” button, it must create the worker record in the database and appear immediately in the UI.
- Use a real database with proper relational structure and migrations.
- Implement authentication (sign up, login, logout) and a secure session/JWT approach.
- Use role-based access control (RBAC) and tenancy isolation (each business/company has isolated data).
- Every create/update/delete must record timestamps AND “who did it” in an audit log.
- Jobs must have unique identifiers (human-readable Job ID like JOB-000123).
- Equipment/Assets must be stored as objects that can be created, updated, deleted, and assigned.
- Workers must have roles:
  - Businesses can create custom role names (e.g., “Senior Installer”, “Crew Lead”)
  - Provide a default dropdown list for common roles for speed (e.g., Owner/Admin, Manager, Supervisor, Crew Lead, Worker, Contractor, Driver)
  - Each role must map to permissions/authority (RBAC). Allow business admins to configure permissions.

Tech Requirements (choose one stack and implement fully):
Option A (preferred): Next.js + TypeScript + Prisma + PostgreSQL + NextAuth
OR
Option B: React + TypeScript + Express + Prisma + PostgreSQL + JWT sessions
Choose ONE and implement properly with a clean repo structure, environment variables, and migrations.

Core Entities and Required Fields:
1) Company (Tenant)
- id, name, createdAt
2) User
- id, companyId, name, email (unique), password hash (if not using OAuth), roleId, createdAt, lastLoginAt
3) Role
- id, companyId, name (custom), permissions JSON, createdAt
4) Worker
- id, companyId, firstName, lastName, phone, email (optional), status (active/inactive), roleTag (dropdown helper), customRoleId (optional), skills (optional), createdAt, updatedAt
5) Equipment/Asset
- id, companyId, name, category, serialNumber (optional), status (available/rented/maintenance/retired), conditionNotes, createdAt, updatedAt
6) Job
- id, companyId, jobId (unique readable like JOB-000123), title, description, status (planned/active/completed/cancelled), priority, startAt, endAt, locationAddress, latitude, longitude, createdAt, updatedAt
7) Assignments
- WorkerAssignment: workerId, jobId, startAt, endAt, notes, createdAt
- EquipmentAssignment: equipmentId, jobId OR workerId (support both), startAt, endAt, notes, createdAt
8) AuditLog
- id, companyId, actorUserId, entityType, entityId, action (CREATE/UPDATE/DELETE/ASSIGN/UNASSIGN/LOGIN), before JSON, after JSON, createdAt

Business Rules:
- Tenancy: A user can ONLY see data for their companyId.
- RBAC: Permissions enforced server-side (not just UI).
- Conflict detection:
  - Worker cannot be assigned to overlapping jobs unless user has override permission.
  - Equipment cannot be assigned to overlapping jobs unless override permission.
  - When conflict occurs, show a clear UI warning and block save by default.
- Job identifiers:
  - Auto-generate sequential JOB-000001 style per company.
- Audit logs:
  - Log every CRUD + assignment change, with before/after snapshot.

UI Requirements:
- Clean, professional admin dashboard with left sidebar navigation:
  - Dashboard
  - Jobs
  - Workers
  - Equipment
  - Assignments (or integrated into Jobs detail)
  - Map (Interactive Jobs Map)
  - Roles & Permissions (admin only)
  - Audit Log (admin/manager)
  - Settings (company profile)
- Each section must support:
  - List view with search/filter
  - Create/Edit/Delete
  - Detail view
- Jobs page:
  - Create job with address and optionally coordinates.
  - Show assigned workers and equipment.
  - Provide “Assign Worker” and “Assign Equipment” actions with conflict checking.
- Workers page:
  - Add/edit worker.
  - Choose role tag from dropdown (common roles).
  - Optional: map to a custom Role (permissions are for app users; worker “role tag” is operational label).
- Equipment page:
  - CRUD assets.
  - Assign to job or worker for time windows.

Interactive Jobs Map Tab (Required):
- Add a dedicated “Map” page that displays all planned/active jobs on an interactive map.
- Use Google Maps OR an alternative (Mapbox/Leaflet with OpenStreetMap) if Google keys are not available.
- Each job is a marker with a unique colour or icon based on status (planned/active/completed).
- Clicking a marker opens a panel with jobId, title, status, dates, assigned workers/equipment, and a link to the job detail page.
- Provide filters on the map (status, date range, priority).
- The map must load jobs from the database, not hardcoded.

Quality & Security:
- Validate inputs (server-side).
- Prevent IDOR (users can’t access other company data by guessing IDs).
- Use parameterized DB queries via ORM.
- Implement robust error handling and user-friendly error messages.
- Provide seed data for local development (one company, admin user, sample workers/jobs/assets).
- Add minimal tests (at least for auth + tenancy + conflict detection).

Deliverables:
1) Working app with all pages and buttons functional.
2) Database schema + migrations.
3) Setup instructions in README (env vars, DB setup, running dev server).
4) Clear code comments for key logic: tenancy checks, RBAC middleware, conflict detection, jobId generation, audit logging.

Start by:
- Proposing the chosen stack (A or B) and the folder structure.
- Creating the Prisma schema (or DB schema) and migrations.
- Implementing auth + tenancy middleware.
- Then build core CRUD pages and the Map page.
